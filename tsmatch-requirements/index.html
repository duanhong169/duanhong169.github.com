3. 详细设计

3.1 Models（Play框架组件）及相互依赖关系

师生配系统一共包括如下一些Model（按生成关系排序）：

  1. User 代表一个用户账号，所有用户注册后都将在系统中生成一条User记录
  2. UserInfo 代表一个账号的用户信息，在用户注册后会生成一个默认的UserInfo（几乎为空），系统将引导用户完善自己的信息，并且根据用户自身的信息完整程度进行权限分配
  3. Post 代表用户发布的内容，用户可以在师生配平台上发布相关信息，该信息将会接收系统审批，信息可以包括附件（UploadFile），信息可以拥有一个或多个标签（Tag），信息在发布后，相关权限的用户可以对其进行评论（Comment）
  4. Tag 用于标识信息或资源的标签，系统将拥有一系列事先定义的或者用户生成的标签，根据这些标签将能够筛选师生配平台的数据
  5. Comment 评论
  6. UploadFile 上传的文件，用户按照相应权限或者积分限制可以上传若干文件
  7. Rating 用户根据自己的权限可以对其他用户或者资源进行评分

上述7种Model的相互关系如下：

  1. A -> B 代表B依赖于A，即当A不存在时，B也随之消失。

    • User -> UserInfo
    • Post -> Comment
    • Post(belongsto) -> UploadFile

  2. A ~> B 代表B与A存在联系，例如某个Post是由某个User所发布，但是与A -> B存在的区别是，当A ~> B时，在A不存在时，B依然存在。

    • User ~> Post（用户的注销不会导致发布信息消失）
    • UserInfo(rater/rated) ~> Rating(评分一旦生效就永远存在)
    • UserInfo(owner) ~> UploadFile（用户的注销不会导致上传文件的消失）
    • \*(whatever) ~> Tag
