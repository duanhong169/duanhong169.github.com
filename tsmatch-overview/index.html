师生配Overview

Apr 24th, 2012

为了方便新的成员加入，这里对师生配的情况做一个简单地说明。

什么是师生配

师生配，即老师和学生之间的配对，或者可以说是提供教育的一方与需要接收教育的一方之间的配对。

目前采用到的技术

鉴于我们对Java都比较熟悉，因此采用了目前比较流行的Web Application开发框架Play framework，Play framework相关信息可以直接到它的主页进行了解，由于在这个项目开始时Play framework 2.0尚未发布，因此我们的开发仍然采用的是Play framework 1.2.4版本，该版本的文档首页在这里，使用Play framework除了需要Java，可能还需要对Groovy语言有简单的了解，因为Play framework的模板引擎（The template engine）使用Groovy作为表达式语言。

Play framework提供了许多的Module，这些Module简单的说就是由其他开发人员所开发的一些可以被我们直接应用到开发中的一些模块，例如一些用于整合其他技术的Module、实现了独立功能的Module以及一些Toolkit类的Module等等，当我们想要实现某个功能时，可以首先到Module的页面搜索一下是否已经有可用的或者类似的Module已经实现了这个功能或者有助于这个功能的实现。目前已经使用到的Module包括：

  1. casino，这个Module不知道出于什么原因，在Play framework的相关页面已经不能访问，该Module的源码地址在casino-play,这个Module所提供的功能包括用户的登录注册、邮件激活账户、找回密码、ReCaptcha验证码、用户角色管理、以及采用hash值方式保存密码等;
  2. crud，这个模块随Play framework一起发布，是一个比较必要的模块，提供了一个用户对JPA Model对象进行管理的Web界面。
  3. elasticsearch，源码地址在playframework-elasticsearch，这个Module用于在Play! Framework应用程序中集成Elastic Search，Elastic Search是一个构建在Apache Lucene之上的开源、分布式、RESTful的搜索引擎，借助于elasticsearch这个Module可以使得我们的每个Model能够被Elastic Search自动索引和更新索引（例如，当某个用户的资料被修改后，Elastic Search能够自动更新对这个资料的索引），借助于Elastic Search的Java API使用起来非常的方便。

在实现一些其他功能时，还用到了如下一些工具或技术：

  1. ajax file upload，文件上传功能ajax实现，支持拖拽上传，源码地址在file-uploader。
  2. Google Static Maps API V2，Google静态地图API，可以根据提供的经纬度数值返回一个包含该经纬度的地图图片。
  3. Google Maps JavaScript API v3，用于实现一些需要与用户交互的地图功能，目前实现的是能够使用户在网页上拖动marker来指定自己的地理位置。

具体实现

在具体的实现方面，目前实现了如下几个Model：

  • User：代表一个用户账号，里面仅仅包括了用户名、密码、邮箱等等账号信息，另外JpaUserManager是casino所需要的；
  • UserInfo：代表了一个用户的详细信息，该Model与User是一一对应的关系（@OneToOne），主要包括了该用户信息所属的User、地址、用户类型、积分、地理坐标、头像等等。
  • UploadFile：代表用户上传的文件。
  • Ratring: 代表一次评分。
  • Comment、Post、Tag：来自于Play framework示例Yabe的遗产Model，还不确定是否有留下的价值。

实现了如下几个Controller：

  • Admin：可以作为用户的“个人主页”入口。
  • Application：整个Web应用程序的入口Controller，其中index()方法的作用就是渲染首页，在这个Controller中还包含了实现搜索功能的方法。
  • UploadFiles、Users、UserInfos、Comments、Posts、Tags：用于操作对应Model的数据。

下一步需要实现的功能

在目前的基础上，亟需实现的功能包括：

  • 评分功能（暂缓，涉及到整个配对流程、交易流程）：同一个用户只能在完成了一次配对之后才能对配对的另一方做出评价。结合积分划分用户等级。

  • 完善用户中心（进行中）：使得用户可以存储更加丰富的资料，主要是图像文件的存储和显示（Gallery）。UserInfo应该包含所有需要的信息，这些信息将被分别展示到“个人中心”或者其他的地方。这里可能需要实现更多的界面。

    1. “我的师生配”界面（用户主界面）——该界面包括资料概览，消息数提示，积分管理，收藏，评价管理，历史数据（已完成的配对、正在进行中的配对）等信息
    2. “我的资料”界面——该界面包括用户的详尽信息，通过多个下级分类显示资料，并且能够对资料进行编辑
    3. “系统设置”界面——设置所需消息提示，所需服务等等，修改头像、密码等等
    4. “高级搜索”界面——使用户能够进行自定义的搜索

  • 将用户匹配的结果以通知的形式发送给正确的用户：这个功能包括了匹配和通知两个功能，其中匹配功能可以基于elasticsearch实现（定期搜索或者事件触发），而通知功能则可能需要借助于其他工具或技术，例如基于XMPP的即时通信，或者使用fire 5等等。

  • 用户自定义：出于用户可能的需要，需要使得用户能够自定义他人是否能够通过搜索或匹配的功能找到自己。
  • 搜索功能增强（深入）：实际上上一条也是搜索功能增强的一部分，这需要较深入地研究Elasticsearch API，包括各种Query创建方式以及Filter创建方式，还包括对ElasticSearch的相关配置（Settings&Mappings）。
  • 用户状态记录与管理：记录指的是日志记录，可以参考知乎的个人主页里对于用户每次编辑的记录形式；管理主要是基于付费功能（即教师和培训机构），即当该付费用户过期后，系统应该能够暂时屏蔽其相关信息，使得其他用户不能够搜索到他，并且当他继续付费后，能够取消掉屏蔽。
  • 各种列表的分页提取数据（paginate），即用户列表，搜索结果列表，以后可能会有的文件管理列表的分页功能。这个功能可能可以借助Play!的一个Module来完成Paginate。
  • 需要进行主要调整的内容：将文件上传功能从编辑资料页面独立出来，重新布局个人资料的展示，登录及注册界面的美化。
  • 各种资源内容可配置，例如目前首页上的6栏广告。

Update：已经实现的功能

  • 评分功能：已登录用户可以对其他用户进行评分，评价信息将被储存，通过所有的评分计算出该用户的最终得分。

  • 邮件系统：直接使用gmail服务器即可。See Using Gmail。

    邮件系统：目前的邮件系统还没有实现，采用的是伪造邮件的方式，参看application.conf文件中的：

    # Mail configuration
# ~~~~~ 
# Default is to use a mock Mailer
mail.smtp=mock

    因此需要实现一个真实的邮件系统。

  • 头像上传：用户已经可以上传一张图片作为自己的头像。

  • 分类搜索结果：已经可以对搜索的结果进行分类展示，即单独显示学员、教员、培训机构的搜索结果。
